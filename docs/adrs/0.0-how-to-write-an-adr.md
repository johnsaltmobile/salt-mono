# How to Write an ADR

In software development, an Architecture Decision Record (ADR) is a document that records a major, consequential decision that will affect the future of the project. When deciding whether a decision requires an ADR, I sometimes ask myself if the decision is a "fork in the path," so to speak. Regardless of the "fork" that I choose, will the project shape up much differntly than if I'd taken the other path?  

An ADR explains 

- Why the decision is necessary—this is a brief "context" or "overview" that oulines the problem or circumstances driving the decision.  
- The decision and rationale behind it.  
- The other options that were considered.   
- The consequences of the choice. What is now possible? What is no longer possible? Are there new pitfalls to be aware of? Pitfalls avoided?

# Example ADR Format

## Title: [Short description of the decision]

### Status
[Proposed, Accepted, Deprecated, Rejected]

### Context
[The problem or need driving the decision.]

### Decision
[The decision made and the reasoning behind it.]

### Alternatives Considered
[Other options and why they were not chosen.]

### Consequences
[The trade-offs and implications of the decision.]
