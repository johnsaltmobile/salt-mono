# ADR 1.0 "Choosing the Expo Workflow"

## Status 
Accepted

## Overview
Because the Salt project will build its mobile client with React Native, we need to choose between the standard [React Native CLI](https://reactnative.dev/docs/getting-started-without-a-framework) workflow and the [Expo](https://docs.expo.dev/get-started/set-up-your-environment/) workflow. 

The primary architectural distinction lies in the management of Native Configurations. The React Native CLI requires engineers to directly own and manipulate the `ios` and `android` project directories as permanent fixtures. Conversely, the Expo workflow utilizes **Continuous Native Generation (CNG)**, treating these native folders as ephemeral artifacts. In this model, the "source of truth" is defined through **Infrastructure as Code (IaC)**—specifically `app.json` and Config Plugins—allowing the native environment to be programmatically generated and version-controlled.

## Decision 
I will use the Expo workflow for both the Salt project and its supporting demos. This choice is driven by **operational efficiency** and **time to market**. 

By speed, I do not mean cutting corners. I mean that development time is significantly compressed through the use of Expo’s curated SDK. This SDK provides a suite of pre-verified, cross-platform libraries (e.g., camera, location, notifications) that are strictly versioned to work together, effectively eliminating the "dependency hell" common in bare CLI projects. Furthermore, utilizing Config Plugins allows us to manage native setup as code, ensuring the build environment is reproducible and transparent across the team.

Expo has matured significantly. Historically, it was viewed as a "junior" or "finicky" sandbox because it forced developers into a "Managed Trap"—if you needed a native feature not in the SDK, you had to "eject," a one-way process that often resulted in a broken, halfway-bare project. That era is over. With the introduction of **Development Builds**, we are no longer limited to a pre-compiled sandbox. We can now create custom development clients that include any native dependency while remaining within the Expo pipeline. 

The reliability of the workflow is further proven by its support for complex, platform-specific features like **iOS Live Activities** (the real-time widgets found on the Lock Screen and in the Dynamic Island). Previously, these required manual Swift implementation in a bare project; now, they can be codified into the Expo prebuild process using TypeScript. This level of maturity is why the React Foundation now officially endorses Expo as the recommended way to build React Native apps.

## Alternatives
### React Native CLI (Bare Workflow)
The bare CLI workflow was rejected primarily due to high maintenance overhead and the **double-edged sword of tribal knowledge**. In a CLI project, manual tweaks to Gradle files or Podfiles often exist only in the memory of the engineer who performed them. If that engineer leaves, the project is effectively paralyzed by lack of documentation. While the CLI offers absolute transparency, the cost is a "non-value-add" labor burden of manual platform upgrades and environment synchronization.

## Consequences
### Positive
* **Infrastructure as Code**: All native configurations are codified in Git, making the environment self-documenting and reproducible.
* **Versioned Stability**: The curated SDK ensures that core native modules are always compatible with the React Native version in use.
* **CI/CD Readiness**: The workflow makes the use of EAS (Expo Application Services) an option for automated cloud builds and distribution, even if we choose to run local builds.

### Neutral/Negative
* **Loss of Native "Grit"**: And now to the other edge of the blade: by abstracting the native layer, there is a risk that engineers lose familiarity with the underlying platform mechanics. 
* **Abstraction Overhead**: If a niche third-party library does not provide a Config Plugin, we must invest time in writing a custom plugin to automate the native changes during the prebuild phase.
* **Deep-Tech Nuances**: We must account for sequential plugin execution. Under the "Last Writer Wins" rule, if multiple plugins target the same native property, the final plugin in the sequence takes precedence.
* **Artifact Size**: There is a negligible increase in binary size (typically 5-10MB) due to the framework's overhead. In the modern era of high-speed connectivity and storage, this is a minor cost of scale when weighed against the massive gain in developer velocity. 
* **Workflow Discipline**: To maintain professional control, we will bypass the generic **Expo Go** app (which limits tool access and prevents native debugging) and instead maintain our own **Development Builds**. This ensures full access to the native debug bridge for monitoring live Firebase events and other deep-level diagnostics.

Config Plugins
These are TypeScript/JavaScript functions that automate the modification of your ios and android project directories during the prebuild process.

They act as a declarative abstraction, replacing manual, direct manipulation of native files with a high-level API.

Explanation: In an imperative approach, you would manually search for specific lines in a complex file like AndroidManifest.xml to add a permission. With declarative abstraction, you simply declare the desired state (e.g., "I need camera access") in your configuration code. The framework then abstracts away the underlying complexity, programmatically injecting the necessary logic into the native project on your behalf.

Native Configurations
These are the platform-specific settings required for an app to function, such as permissions (camera, location), app icons, splash screens, and bundle identifiers.

In traditional workflows, these settings reside in boilerplate-heavy files like Info.plist, AndroidManifest.xml, Podfiles, and build.gradle.

Version-Controlled Build Environment
Under CNG, the ios and android folders are ephemeral artifacts that are not committed to Git.

The "source of truth" for your native setup is stored entirely as code within your repository—primarily via app.json or custom Config Plugins.

This ensures that any developer can recreate the exact same native environment by running a single command, which eliminates "tribal knowledge" and reduces manual configuration errors across the team.

The following summary clarifies the hierarchy of the Expo configuration system, explaining how the static manifest and dynamic plugins co-exist within a single "source of truth," sir.

The Manifest vs. The Logic
It is helpful to view the "source of truth" not as two competing files, but as a single pipeline.

app.json (The Manifest): This is the high-level declaration of your app's identity. It contains static values like the name, slug, and version.

Config Plugins (The Logic): These are functions referenced inside the plugins array of your app.json. They are the workers that execute specialized tasks (like adding a specific line to a Podfile) that a simple JSON key cannot describe.

Execution Order and Precedence
In the Expo workflow, there is a strict sequence of operations during the prebuild phase that dictates precedence:

Static Evaluation: Expo first reads the static fields in your app.json.

Plugin Iteration: Expo then iterates through the plugins array from top to bottom.

The "Last Writer Wins" Rule: If two plugins attempt to modify the same native property (e.g., the app's orientation), the plugin listed last in the array will generally overwrite the changes made by the previous one.

Resolving Contradictions
Technically, a contradiction is resolved by this sequential execution. Because Config Plugins are essentially functions that receive the current configuration, modify it, and return the new version, they are cumulative. If your app.json defines a generic setting but a plugin modifies that same setting for a specific native edge-case, the plugin’s logic will prevail in the final generated native code.

THE WORKFLOW ARGUMENT
RESPONSE #1
The following points outline why the Expo framework remains the superior architectural choice for modern mobile development in 2026.

Continuous Native Generation (CNG): Automating the creation of the ios and android directories via npx expo prebuild ensures your native code remains a predictable artifact rather than a source of technical debt.

Expo Router Integration: This provides a file-based routing system that unifies web and native navigation patterns while utilizing native platform APIs for performance.

Curated SDK Ecosystem: Every library within the Expo SDK is rigorously tested for compatibility, effectively eliminating "dependency hell" during upgrades.

Advanced Module API: The Expo Modules API offers a modern, Swift/Kotlin-first approach to writing native code, which is more ergonomic than legacy CLI methods.

Live Activities and Widgets: The recently matured @expo/widgets package allows for the seamless implementation of iOS Live Activities using familiar React patterns.

Unified Development Workflow: Features like "Development Builds" provide the flexibility of custom native code while retaining the rapid iteration of Expo Go.

Infrastructure as Code: Utilizing Config Plugins allows all native configurations to be defined in TypeScript, ensuring your build environment is version-controlled.

First-Class Web Parity: Expo’s commitment to "Universal React" ensures that the same codebase can target mobile and web with minimal platform-specific branching.

Simplified Upgrade Cycles: The npx expo install --fix command automates the task of aligning package versions, a major pain point in bare CLI projects.

Industry Standard Recommendation: As of 2026, the official React Native documentation and major industry players recommend Expo as the default for production.


RESPONSE #2
The following arguments demonstrate why the React Native CLI remains the essential choice for engineers requiring absolute transparency and control over their mobile architecture.

Architectural Transparency: Direct ownership of the ios and android directories eliminates the "black box" of CNG, ensuring every line of native code is explicit and auditable.

Minimal Binary Footprint: By avoiding the overhead of the Expo SDK, CLI-based projects can achieve smaller bundle sizes, critical for performance-sensitive applications.

Zero-Day Native Parity: You are never at the mercy of an SDK alignment cycle; new iOS or Android features can be implemented directly without waiting for a framework update.

Granular Dependency Control: In a CLI project, you ship only the code you explicitly include, avoiding the "framework tax" of carrying unused native modules.

Direct Build Lifecycle Management: Manual control over Podfiles and Gradle scripts allows for highly specialized build configurations often restricted by Config Plugins.

Superior Brownfield Integration: For existing native applications, the CLI is the only viable path where React Native must coexist as a guest within a legacy codebase.

No Abstraction Overhead: While Config Plugins attempt to simplify native changes, editing the native files directly is often faster and more reliable for experienced engineers.

Elimination of Vendor Lock-in: Relying on Expo’s proprietary abstractions creates a dependency on their specific tooling; the CLI uses standard industry tools.

Simplified CI/CD Flexibility: Without Expo-specific CLI wrappers, your project can more easily integrate into bespoke or high-security internal build pipelines.

Pure Platform Ownership: Owning the native code ensures that long-term maintenance is not tied to a third-party framework, providing a stable foundation for the future, sir.
